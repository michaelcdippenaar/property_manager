## Goal
Replace every plaintext / default-styled email with a branded, mobile-friendly, accessible HTML template — content sourced from `content/emails/` so copy edits don't require code changes.

- [x] Base layout: Klikk brand (Navy #2B2D6E, Accent #FF3D7F, Bricolage Grotesque fallback → system serif; DM Sans fallback → system sans), footer with unsubscribe + info officer contact
- [x] Templates for: welcome (agent + tenant), mandate sent, mandate signed, lease sent for signing, lease signed, rent reminder, rent received, rent overdue, maintenance update, invite tenant, password reset, 2FA enrollment, 2FA recovery code reissue
- [x] Plaintext fallback for every HTML
- [x] Content sourced from `content/emails/*.md` (markdown + frontmatter)
- [x] Accessibility: alt text on images, sufficient contrast, no text-in-image
- [x] Deliverability: SPF + DKIM + DMARC set on sender domain; warmup schedule documented

### 2026-04-22 — reviewer: changes requested

Two bugs must be fixed before this can pass to testing:

**1. [BLOCKING] Double-escaping of `body_text` HTML — `backend/apps/notifications/services/email.py`**

`_simple_md_to_html()` produces HTML strings like `<p>Your lease...</p>`. This string is stored in `render_ctx["body_text"]` and then passed through `_render_string()` which calls `Template.render(Context(context))`. Django's `Context` autoescapes by default, so every `<` and `>` in `body_text` is escaped to `&lt;` and `&gt;`. The final email HTML will contain literal text `<p>Your lease...</p>` instead of rendered paragraphs — every multi-paragraph email body is broken.

Fix: after computing `body_text`, wrap it with `mark_safe()` so Django does not re-escape the already-trusted HTML output:

```python
from django.utils.safestring import mark_safe
# ...
render_ctx["body_text"] = mark_safe(_simple_md_to_html(raw_body_rendered))
```

Note: `body_text` is constructed from content in `content/emails/*.md` which is developer-controlled, not user-supplied. `mark_safe` is appropriate here. The same applies to the intermediate assignment at line 198 (`"body_text": _simple_md_to_html(md["body"])`), which is later overwritten at line 211 — only line 211's assignment needs the `mark_safe` wrap since that is what actually enters the final render context.

**2. [BLOCKING] Zombie `services.py` not deleted — `backend/apps/notifications/services.py`**

`services.py` still exists on disk alongside the new `services/` package directory. Python resolves the package over the module at runtime so production behaviour is correct today, but:
- Any `grep` or IDE search for `send_email` finds two definitions and cannot tell which is live
- A developer editing `services.py` thinking they are changing live code will be silently wrong
- The existing test suite (`test_hub/notifications/unit/test_services.py`) patches `apps.notifications.services.*` — these patches still work because Python loads `services/__init__.py`, but the situation is fragile and confusing

Delete `backend/apps/notifications/services.py`. The file's content is already fully replicated in `backend/apps/notifications/services/__init__.py` (the diff shows identical code).
