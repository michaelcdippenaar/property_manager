## Goal

Deliver a channel-abstracted OTP service backed by email (AWS SES, already configured) that unblocks registration, password reset, and sensitive-change flows at launch, with SMS swappable in as a zero-rework drop-in adapter once WinSMS/Panacea onboarding clears.

## Acceptance criteria

- [x] `apps/accounts/otp/service.py` exposes `OTPService.send(user, purpose: str)` and `OTPService.verify(user, code: str, purpose: str) -> bool`
- [x] `apps/accounts/otp/channels/base.py` defines a `Channel` ABC with `send(recipient, code, context) -> None`
- [x] `apps/accounts/otp/channels/email.py` implements the email channel using the existing Django email backend (SES in prod, console in dev); template at `apps/accounts/templates/otp/email.txt` — plain text, code prominent, "expires in 5 min", POPIA footer
- [x] `apps/accounts/otp/channels/sms.py` is a stub that raises `NotImplementedError("SMS channel pending WinSMS/Panacea integration")` — interface-compatible so the provider can be wired later without touching callers
- [x] Channel selection driven by `settings.OTP_CHANNELS` (ordered list; first-working wins); default `["email"]` for v1
- [x] 6-digit numeric code, 5-minute TTL, hashed at rest — never store plaintext; HMAC-SHA256 with per-user+purpose salt (keyed on `user_pk:purpose:SECRET_KEY`)
- [x] Rate limits: max 5 OTP issues per user per hour; max 3 verify attempts per code (lock after 3 fails, require new code)
- [x] Per-purpose code scoping: a code issued for `purpose="registration"` cannot verify a `purpose="password_reset"` request — prevents cross-flow replay
- [x] `OTPCodeV1` model (Django name; `verbose_name = "OTP Code"`): `user`, `purpose`, `code_hash`, `created_at`, `expires_at`, `consumed_at`, `attempt_count`, `channel_used` — migration 0019 included; `db_table = "accounts_otp_code"`
- [x] `OTPAuditLog` rows written on every issue / verify-success / verify-fail (POPIA audit trail) — new model (no existing audit app for OTP events); piggybacked into `accounts` app
- [x] Unit tests cover: service send/verify, email channel, rate limiting, replay prevention, TTL expiry, cross-purpose isolation — 45 tests, 95%+ coverage on the `otp` package

## Handoff notes

### 2026-04-23 — implementer

Implemented the full OTP service package under `backend/apps/accounts/otp/`.

**Key decisions:**

1. **Model naming — `OTPCodeV1` not `OTPCode`.** A legacy `OTPCode` placeholder already existed in `accounts/models.py` (stores plaintext code, no purpose scoping). Renaming it would break existing views and test_hub tests that import `OTPCode`. To avoid that scope creep, the new full-featured model was named `OTPCodeV1` (with `verbose_name = "OTP Code"`, `db_table = "accounts_otp_code"`). Both models live in the `accounts` app. Migration 0019 only creates two new tables — no existing table is touched.

2. **HMAC-SHA256 keyed on `user_pk:purpose:SECRET_KEY`.** Provides per-user, per-purpose domain separation. Compared to `make_password`/`check_password`, this is faster (no bcrypt rounds needed for short-lived codes) and still cryptographically sound.

3. **POPIA audit trail on `OTPAuditLog`** (new model, not piggybacked on `AuthAuditLog`) because OTP events have distinct metadata (purpose, channel, otp_code_id) that doesn't fit the existing `AuthAuditLog` schema cleanly.

4. **`otp/__init__.py` is empty** (no top-level imports) to avoid circular imports when `accounts/models.py` imports from `otp/models.py` during Django startup.

**Test results:** 45 tests, all pass. Coverage on the `otp` package: `service.py` 96%, `models.py` 94%, `channels/email.py` 100%, `channels/sms.py` 100%, `channels/base.py` 83% (abstract method body unreachable by design).

**Caveats for reviewer:**
- The old OTP views (`OTPSendView`, `OTPVerifyView` in `accounts/views.py`) still use the legacy plaintext model. Wiring the new service into views is explicitly out of scope per the task.
- The `test_hub` integration tests for `OTPCode` continue to test the legacy model and remain unaffected.
- `_pick_recipient` currently only handles `email` and `sms` — adding a new channel requires a one-line addition there plus a new `channels/<name>.py`.

### 2026-04-23 — reviewer

Review passed.

Checked against all acceptance criteria:

- `OTPService.send(user, purpose)` and `OTPService.verify(user, code, purpose) -> bool` present with correct signatures in `service.py`.
- `Channel` ABC in `channels/base.py` with abstract `send(recipient, code, context) -> None`. `EmailChannel` and `SMSChannel` both extend it correctly.
- Codes hashed at rest via HMAC-SHA256 keyed on `user_pk:purpose:SECRET_KEY`; `_generate_code` uses `secrets.randbelow`; plaintext never persisted.
- Rate limit: 5 issues/hr enforced in `send` with `OTPRateLimitExceeded`. Verify lock: `attempt_count >= OTP_MAX_ATTEMPTS` raises `OTPMaxAttemptsExceeded`.
- TTL expiry: `expires_at__gt=now` filter in `verify`; `OTP_CODE_TTL_SECONDS=300` in `base.py`.
- Per-purpose scoping: `purpose` is both a DB filter column and baked into the HMAC key — cross-flow replay impossible by construction.
- Audit log: `OTPAuditLog` rows written on every issue/verify-success/verify-fail/rate-limit/lock path.
- Migration 0019: additive only — creates `accounts_otp_code` and `OTPAuditLog` tables; legacy `OTPCode` model (`accounts_otpcode`) is untouched (different table name, no collision).
- 45 tests (32 service + 13 channel), all scenarios required by the AC are covered.
- No raw SQL, no PII logged, no auth endpoints introduced (service layer only).
