## Goal
Fix the role-alias mismatch that causes `extract_signer_fields` to return 0 signing fields for a tenant signer, making it impossible to complete native signing for any template where the HTML uses `data-signer-role="tenant"`.

## Acceptance criteria
- [x] Either normalise template HTML roles to `tenant_1` / `tenant_2` during `_convert_legacy_signing_span`, OR extend `extract_signer_fields` to treat `"tenant"` as an alias for `"tenant_1"` (and `"landlord"` → `"landlord_1"` etc.) — **both** applied
- [ ] `esigning_native_signing_test` Step 5 returns > 0 fields for any template that has a tenant signature block (manual E2E — for tester)
- [ ] Template 57 (and any other template with `data-signer-role="tenant"`) produces signing fields when a `tenant_1` signer creates a native submission (manual — for tester)
- [x] No regression in existing `tenant_1` / `landlord_1` explicit-role templates — covered by `test_no_cross_role_contamination` + `test_explicit_tenant_1_role_matched`
- [x] Regression test added: `test_extract_signer_fields_tenant_alias` — asserts fields are found when signer record role is `tenant_1` but HTML uses `data-signer-role="tenant"`

## Files likely touched
- `backend/apps/esigning/services.py` (`_convert_legacy_signing_span` ~line 437, `_role_to_tiptap` ~line 554, `extract_signer_fields` ~line 597)

2026-04-23 — Review requested changes (rentals-reviewer).

**Blocker: asymmetric alias maps cause a landlord-path regression.**

`_role_to_tiptap` (services.py:571-577, used by `create_native_submission` to populate `signer_records[].role`) maps:
- `landlord → landlord` (no `_1` suffix)
- `lessor → landlord`
- `tenant → tenant_1`

The new `_legacy_role_alias` (services.py:450-456) and `_html_role_aliases` (services.py:618-624) maps, however, both canonicalise:
- `landlord → landlord_1`
- `lessor → landlord_1`

This means:

1. For any legacy template (`<span data-type="signature-block" data-signer-role="landlord">`), `_convert_legacy_signing_span` now emits HTML with `role="landlord_1"`. But the signer record still stores `role="landlord"` (via `_role_to_tiptap`). `extract_signer_fields(html, "landlord")` then compares HTML canonical `"landlord_1"` against signer_role `"landlord"` → 0 fields. **This is a new regression introduced by the fix**: every landlord-signing flow against a legacy template that was working before this patch now breaks.

2. Confirmed by searching the codebase: `landlord_1` exists only in this PR + its tests; `SignatureBlockNode.ts` (admin TipTap) uses `'landlord'` (line 79, 84, 89, 94, 107). The canonical landlord tiptap role in the rest of the system is unsuffixed.

**Required fixes (choose one internally consistent approach):**

1. Align the maps. Either:
   - **Option A (recommended, smaller blast radius):** change both `_legacy_role_alias` and `_html_role_aliases` so `landlord → landlord` and `lessor → landlord` (keep them unsuffixed to match `_role_to_tiptap` + `SignatureBlockNode.ts`). The entries become no-ops for `landlord` but `lessor → landlord` still adds value.
   - **Option B:** also change `_role_to_tiptap` to `landlord → landlord_1` / `lessor → landlord_1` and migrate any stored signer records. This is a much bigger change — do not pick this without PM sign-off.

2. Deduplicate the two alias maps. They encode the same concept and must stay in lock-step — extract a single module-level constant (e.g. `_HTML_ROLE_CANONICAL: dict[str, str]`) and reference it from both `_convert_legacy_signing_span` and `extract_signer_fields`. As written, the next person to add a role (`witness_1`? `agent`?) will update one and forget the other.

3. Add a regression test for the landlord path mirroring `test_extract_signer_fields_tenant_alias` — i.e. signer record role is whatever `_role_to_tiptap` produces for `"Landlord"` input, HTML role is `"landlord"` legacy — fields must be found. The current `test_landlord_alias_matched` tests `landlord_1` as the signer_role, which is **not what the production caller passes**; this test passes the unit but doesn't prove the end-to-end path works. Fix the test to exercise the actual `create_native_submission` → `extract_signer_fields` contract.

4. Minor: `owner` (rental-mandate signing per `_role_to_tiptap`) is absent from both alias maps. Confirm no legacy mandate templates use `data-signer-role="owner"`; if any do, add a passthrough entry or a comment explaining why none is needed.

**What I verified as OK:**
- Case-insensitive comparison (`.lower()` on both sides) — fine.
- No security impact: HTML is server-generated, role comparison is not an auth boundary (link resolution happens upstream in `_resolve_link`), no user-supplied role reaches this code path.
- Multi-role rows: n/a — each tag has a single `role=` attribute, regex handles them independently.
- Whitespace: raw role values are emitted by the same code, no external whitespace risk in practice; not worth adding `.strip()` unless existing templates exhibit it.
- `_convert_legacy_signing_span` alias map correctly handles `tenant`, `co-tenant`, `cotenant`. No third caller emits raw short-form `role=` — grep confirms only the two added sites touch role normalisation.
- Tests for tenant/no-cross-contamination paths are well-constructed.

Please re-submit once maps are reconciled and the landlord regression test is added.
