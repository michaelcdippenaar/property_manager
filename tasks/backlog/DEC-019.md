## Decision question

Should Klikk replace the current Google Authenticator / TOTP second-factor (`UserTOTP` / `TOTPRecoveryCode`, migration 0016) with a pluggable OTP provider — **email OTP in dev/staging** (zero cost) and **SMS OTP in production** (gateway cost)?

## Context

The current TOTP implementation requires users to enrol a TOTP app (Google Authenticator, Authy). This creates friction at onboarding and adds support burden. Email OTP and SMS OTP are simpler UX and better fit the SA market. Migration 0016 introduced `UserTOTP` and `TOTPRecoveryCode`; those models would be deprecated by RNT-SEC-034.

## Options

### Option A — Email OTP everywhere (dev + prod)
- **Pros:** Zero SMS gateway cost; no provider dependency; works offline with email client.
- **Cons:** Email delivery latency; email compromise = 2FA bypass; less secure than TOTP.
- **Recommended for:** dev/staging only.

### Option B — Email OTP (dev/staging) + SMS OTP (prod)
- **Pros:** Stronger security in prod (separate channel); familiar UX for SA users; no app install required.
- **Cons:** Per-message gateway cost; SMS delivery can fail (coverage gaps, spam filters); need a ZA-capable provider.
- **Recommendation:** This is the proposed approach. Needs provider and cost decisions below.

### Option C — Keep TOTP
- **Pros:** No migration work; no ongoing cost; highest security.
- **Cons:** Poor onboarding UX; requires authenticator app; harder to support.

## Questions for MC

1. **SMS provider** — Which gateway for ZA? Preferred candidates: Clickatell, BulkSMS, Twilio. Consider ZA termination rates and SMPP vs REST API support.
2. **Per-message budget cap** — What is the maximum acceptable cost per SMS (ZAR)? Should a monthly cap be enforced (e.g., cut over to email fallback after N messages)?
3. **Fallback behaviour** — If SMS delivery fails, should the system automatically fall back to email OTP, or surface an error and require the user to request an email code manually?
4. **OTP parameters** — 6-digit code? Validity window 5 minutes? (These are the defaults; confirm or override.)
5. **Rate limits** — Max resend attempts per user per hour? Max failed verifications before lockout?
6. **Grace period** — Any existing TOTP-enrolled users must be migrated. Should they be force-reset on next login, or given a transition window (e.g., 30 days)?

## Recommendation

Option B. Use BulkSMS or Clickatell for ZA (both have REST APIs and competitive ZA rates). Default OTP: 6 digits, 5-minute window. Fallback to email on SMS failure. Force-reset existing TOTP users on next login (there are very few in prod at this stage).

## Files likely touched

- `backend/apps/accounts/models.py` (deprecate UserTOTP, TOTPRecoveryCode)
- `backend/apps/accounts/otp/` (new directory — providers.py)
- `backend/config/settings/*.py` (OTP_PROVIDER env var)
