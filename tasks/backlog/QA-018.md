## Goal
Fix the role-alias mismatch that causes `extract_signer_fields` to return 0 signing fields for a tenant signer, making it impossible to complete native signing for any template where the HTML uses `data-signer-role="tenant"`.

## Acceptance criteria
- [ ] Either normalise template HTML roles to `tenant_1` / `tenant_2` during `_convert_legacy_signing_span`, OR extend `extract_signer_fields` to treat `"tenant"` as an alias for `"tenant_1"` (and `"landlord"` → `"landlord_1"` etc.)
- [ ] `esigning_native_signing_test` Step 5 returns > 0 fields for any template that has a tenant signature block
- [ ] Template 57 (and any other template with `data-signer-role="tenant"`) produces signing fields when a `tenant_1` signer creates a native submission
- [ ] No regression in existing `tenant_1` / `landlord_1` explicit-role templates
- [ ] Regression test added: `test_extract_signer_fields_tenant_alias` — asserts fields are found when signer record role is `tenant_1` but HTML uses `data-signer-role="tenant"`

## Files likely touched
- `backend/apps/esigning/services.py` (`_convert_legacy_signing_span` ~line 437, `_role_to_tiptap` ~line 554, `extract_signer_fields` ~line 597)
- `backend/apps/esigning/tests/test_services.py`
