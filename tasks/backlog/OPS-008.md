## Goal
Know within 2 minutes when production is down. Before first client, we need external uptime checks, deep healthchecks, and an on-call path.

## Decision baked in (DEC-004 answered 2026-04-22)
Use **UptimeRobot** (free tier, 50 monitors) as the primary uptime monitor. Complement with **Healthchecks.io** (free tier) for Celery beat + backup success pings. Do NOT configure BetterStack for v1.0 — revisit post-revenue.

## Acceptance criteria
- [ ] `/health/` endpoint: basic liveness (DB connect, Redis connect, Gotenberg reachable, disk space)
- [ ] `/health/deep/` endpoint (auth-gated): runs a full lease-render probe every 15 min
- [ ] UptimeRobot: external uptime monitor hitting every surface (admin, tenant web, backend `/health/`) every 5 min; alerts SMS to MC + email to ops list when down > 2 min
- [ ] Healthchecks.io: ping URLs configured for Celery beat heartbeat and backup success jobs; alert on missed ping
- [ ] Status page: public `status.klikk.co.za` via UptimeRobot status page feature (included in free tier)
- [ ] On-call rotation doc in `docs/ops/on-call.md` (v1.0 = MC solo, named)
- [ ] Do NOT add BetterStack — deferred to post-revenue

## Files likely touched
- `backend/apps/health/views.py`
- `backend/config/urls.py`
- `docs/ops/on-call.md` (new)

## Test plan
**Manual:**
- Stop DB container → /health returns 503 → UptimeRobot alert fires within 2 min
- Stop Gotenberg → deep probe fails → alert
- Pause Celery beat → Healthchecks.io misses ping → alert fires
