## Goal
Fix the `drill_reminder` Docker service so it actually runs `drill-reminder.sh` on the 1st of each month instead of running `backup.sh` at midnight (which doubles S3 writes and silently never sends the reminder).

## Acceptance criteria
- [ ] `drill_reminder` service uses a dedicated crontab file (e.g. `/etc/crontab.drill`) that schedules `drill-reminder.sh` at `0 8 1 * *`
- [ ] `CRONTAB_CONTENT` env var removed from `docker-compose.backup.yml` (or the Dockerfile entrypoint correctly consumes it)
- [ ] `backup.sh` no longer runs inside the `drill_reminder` container
- [ ] The main backup container still runs `backup.sh` at `0 0 * * *` (unchanged)

## Files likely touched
- `deploy/docker-compose.backup.yml`
- `deploy/Dockerfile.backup`
- `scripts/drill-reminder.sh` (or new `deploy/crontab.drill`)

## Test plan
**Manual:**
- Inspect running container: `docker exec drill_reminder supercronic --list` — confirm only `0 8 1 * *` entry for `drill-reminder.sh`
- Confirm backup container schedule unchanged

**Automated:**
- OPS-003 AC-5 verified in post-deploy smoke check
