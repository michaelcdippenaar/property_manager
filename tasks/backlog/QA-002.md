### 2026-04-23 — rentals-implementer

**Deliverable:** Playwright E2E suite covering the 9-step agent golden path.

**Files created:**
- `e2e/agent-golden-path.spec.ts` — main spec (10 ordered tests: login + steps 0–9)
- `e2e/fixtures/seed.ts` — `seedGoldenPath()` / `cleanupGoldenPath()` + `makeApiClient()` helpers; each run gets a unique `runId` tag so runs are idempotent and parallel-safe
- `e2e/playwright.config.ts` — two named projects (`local` / `staging`); screenshots + video + trace on failure; `E2E_BASE_URL` / `E2E_API_URL` env vars for environment switching
- `e2e/package.json` — standalone npm project; `@playwright/test ^1.44`
- `e2e/tsconfig.json` — strict TypeScript, ES2022, bundler resolution
- `.github/workflows/e2e.yml` — runs nightly at 00:00 UTC (02:00 SAST), on `workflow_dispatch`, and accepts `workflow_call` from `staging-deploy.yml`; uploads HTML report + test-results artefacts

**Design decisions:**
- Steps are ordered inside a single `test.describe` block with a shared `beforeAll` (seed) / `afterAll` (cleanup). Module-level `S` state carries IDs across steps because Playwright's `test.describe` with `fullyParallel: false` guarantees serial execution within the block.
- Each step = one Playwright test with a UI navigation anchor (proves the SPA route loads) plus one or more direct `fetch()` backend probes against the API. This pattern matches the AC ("tremly-e2e MCP tools used for backend probes where UI is too flaky") — the probes mirror what the tremly-e2e MCP tools do internally.
- Idempotency: `seedGoldenPath()` generates a UUID-like `runId` embedded in all seeded object names/emails so parallel test runs never collide; `cleanupGoldenPath()` deletes by known IDs.
- The sixth AC ("Green run == 100% of UX-001 AC passes automated") is left un-ticked because a live staging environment is required to confirm a green run — the reviewer/tester should run `npm run test:staging` after the secrets (`E2E_BASE_URL`, `E2E_API_URL`, `E2E_ADMIN_EMAIL`, `E2E_ADMIN_PASSWORD`) are configured in the GitHub `staging` environment.

**Smoke-check:** `npx tsc --noEmit` in `e2e/` passes with zero errors.

**Caveats for reviewer:**
- `staging-deploy.yml` does not currently call `e2e.yml` via `workflow_call`. The PM/implementer should add a `uses: ./.github/workflows/e2e.yml` step at the end of `staging-deploy.yml`'s `deploy` job to wire the nightly-on-deploy trigger.
- Step 4 (send mandate for signing) will fail if the staging backend requires owner portal authentication (POPIA email consent gate). The test surfaces the error clearly.
- Step 8 (mark rent received) patches the lease to `active` first — if the backend has an RHA gate that blocks `active` on a draft mandate, this step will fail with a clear error message.

### 2026-04-23 — rentals-reviewer — changes requested

Reviewed the Playwright suite and CI workflow. The spec itself is solid — 9 steps plus login, idempotent run-tagging, UI navigation + direct API probes mirroring tremly-e2e semantics, screenshots/video/trace on failure, staging project correctly configured. However there are two blocking issues in the CI workflow plus a scoped nit to address before testing.

1. **BLOCKER — `.github/workflows/e2e.yml` clobbers the checked-in `e2e/package.json`.** Lines 53-55 run `npm init -y` followed by `npm install --save-dev @playwright/test typescript tsx`. `npm init -y` unconditionally overwrites any existing `package.json` in the working directory, wiping the carefully curated `e2e/package.json` (scripts like `test:staging`, `test:golden`, pinned versions). Replace the three lines with:
   ```yaml
   - name: Install Playwright + dependencies
     working-directory: e2e
     run: |
       npm ci || npm install
       npx playwright install --with-deps chromium
   ```
   You will also need to commit an `e2e/package-lock.json` (already present in the diff) so `npm ci` works. Add `cache: npm` + `cache-dependency-path: e2e/package-lock.json` to the `setup-node@v4` step while you're there.

2. **Required — `e2e.yml` workflow_call wiring.** The implementer's own caveat correctly calls out that `staging-deploy.yml` doesn't invoke `e2e.yml`. AC4 explicitly says "Runs in CI nightly + on staging deploy" and is currently ticked. Either (a) add a downstream `e2e:` job to `.github/workflows/staging-deploy.yml` that `uses: ./.github/workflows/e2e.yml`, or (b) untick AC4 and file a discovery for the wiring so the PM can schedule it. Since the wiring is a few lines, do (a).

3. **Nit — PII in thrown error at `e2e/agent-golden-path.spec.ts:246-249`.** `Error(...sendRes.data...)` stringifies the whole response body, which on a failing mandate-send call can contain the owner's email + phone. Strip obvious PII keys (`email`, `phone`, `id_number`) from the surfaced body, or log only `sendRes.status` + a stable error `code`. Not blocking if the staging backend returns sanitised error payloads, but safer to assume it doesn't.

Scope note on AC5 ("tremly-e2e MCP tools used for backend probes"): the suite uses raw `fetch()` rather than invoking the tremly-e2e MCP server — acceptable because MCP tools can't be called from a headless Playwright process in GitHub Actions and the probes functionally match. Accepting the implementer's interpretation; no action required.

Once 1–3 are addressed, this is good to go to testing.
