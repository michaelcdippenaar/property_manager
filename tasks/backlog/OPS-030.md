## Goal
`deploy/backup.sh` must never silently no-op in production. Missing infrastructure credentials in a production environment must be fatal (not skipped), and operators must receive an alert any time a scheduled backup does not complete.

## Acceptance criteria
- [ ] When `DEPLOY_ENV=production` (or `DJANGO_SETTINGS_MODULE` contains `production`), missing infra vars call `die()` not `skip()`; the container exits non-zero and any process manager / cron wrapper surfaces the failure
- [ ] `skip()` helper in non-prod still executes but POSTs a warning to `SLACK_WEBHOOK_URL` if that var is set
- [ ] A UptimeRobot heartbeat URL (or equivalent dead-man's switch) is pinged on every successful S3 upload; the monitor alerts if no ping arrives within ~26 hours
- [ ] `docs/ops/backup-restore.md` updated to document the production guard and dead-man's switch setup
- [ ] Manual smoke-test: run `DEPLOY_ENV=production ./backup.sh` with one required var unset — confirm non-zero exit and no silent skip

## Files likely touched
- `deploy/backup.sh` (lines ~42–48 `skip()`, ~95–101 `infra_vars` loop)
- `docs/ops/backup-restore.md`
- `.env.production.example` (heartbeat URL var)

## Test plan
**Manual:**
- `DEPLOY_ENV=production SLACK_WEBHOOK_URL=<hook> ./backup.sh` with `S3_BUCKET` unset → expect `die()` + non-zero exit
- `DEPLOY_ENV=staging SLACK_WEBHOOK_URL=<hook> ./backup.sh` with `S3_BUCKET` unset → expect Slack warning + exit 0

**Automated:**
- Shell unit test (bats or inline) asserting exit code 1 for production + missing var
