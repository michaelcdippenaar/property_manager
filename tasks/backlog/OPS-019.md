## Goal
Fix the `drill_reminder` container so it fires `drill-reminder.sh` monthly (1st of month, 08:00) instead of running `backup.sh` nightly alongside the regular backup container.

## Acceptance criteria
- [ ] `drill_reminder` service uses its own crontab (`/etc/crontab.drill`) scheduling `drill-reminder.sh` at `0 8 1 * *`, OR the Dockerfile entrypoint writes the crontab from `CRONTAB_CONTENT` before invoking supercronic
- [ ] `backup.sh` no longer runs inside the `drill_reminder` container
- [ ] `CRONTAB_CONTENT` env var removed from `docker-compose.backup.yml` if the dedicated-crontab approach is chosen
- [ ] `docker compose -f docker-compose.backup.yml up drill_reminder` shows only the drill schedule in supercronic's log output

## Files likely touched
- `deploy/docker-compose.backup.yml`
- `deploy/Dockerfile.backup`

## Test plan
**Manual:**
- `docker compose -f deploy/docker-compose.backup.yml config` — verify drill_reminder crontab schedule is `0 8 1 * *`
- Trigger a manual run of `drill-reminder.sh` inside the container and confirm no `backup.sh` execution
