## Goal
Move from `.env.secrets`-file-on-disk to a managed secrets store with audit, rotation, and per-env separation. This is the durable fix — RNT-SEC-001 rotates, OPS-009 ensures it never regresses.

## Acceptance criteria
- [ ] ~~Choose one: AWS SSM Parameter Store (af-south-1) OR 1Password Connect OR HashiCorp Vault Cloud~~ **RESOLVED: AWS SSM Parameter Store af-south-1** (DEC-003 + DEC-020 answered 2026-04-24)
- [ ] Runtime: app reads secrets at boot via SDK, not from `.env.secrets` on disk
- [ ] Local dev: `.env.example` committed (no secrets), `.env.local` gitignored, docs explain how to pull
- [ ] Staging + production: separate paths/namespaces, separate IAM
- [ ] Rotation policy: JWT signing key every 90d, DB password every 180d, third-party API keys on breach
- [ ] Audit log of who read/wrote which secret
- [ ] Emergency break-glass procedure documented

## Files likely touched
- `backend/config/secrets.py` (SDK wrapper)
- `backend/config/settings.py`
- `deploy/docker-compose.prod.yml` (IAM role / token injection)
- `docs/ops/secrets.md` (new)

## Test plan
**Manual:**

- Audit log shows the rotation event

DEC-020 answered 2026-04-24: production is **AWS af-south-1**. Combined with DEC-003 (recommended AWS SSM Parameter Store + KMS af-south-1), the vault choice is confirmed.

Implementer should use:

- KMS key in af-south-1 for parameter encryption
- IAM role attached to EC2 instance profile — no static AWS credentials in the app container
- Rotation schedule as per AC: JWT key every 90d, DB password every 180d, third-party on breach
