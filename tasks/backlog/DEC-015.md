## Question
The staging server is on a private LAN (`192.168.1.235`). GitHub Actions runners cannot reach it, so the SSH deploy step in `staging-deploy.yml` always times out. How should we make staging reachable?

## Options

**Option A — Router port-forward + firewall rule**
- Pros: free, no extra services, immediate
- Cons: exposes a home IP, NAT hairpin issues, IP may change (DDNS needed), security surface on home network
- Recommendation: viable short-term only

**Option B — Cloudflare Tunnel (cloudflared)**
- Pros: free tier, no open port on home router, traffic tunnelled through Cloudflare, stable hostname
- Cons: adds a Cloudflare dependency, tunnel daemon must stay running on the server
- Recommendation: best balance of security and cost for a home-lab staging box

**Option C — Tailscale**
- Pros: zero-trust, no open ports, GitHub Actions Tailscale action available, very simple
- Cons: Tailscale account required, team needs access to the tailnet
- Recommendation: excellent if the team already uses Tailscale

**Option D — Migrate staging to a cloud VM (e.g. Hetzner CX11 ~€4/mo)**
- Pros: public IP natively, eliminates home-lab constraints, matches production topology
- Cons: monthly cost, more infra to manage, migration effort
- Recommendation: right long-term choice if the staging server needs to stay stable

**PM recommendation:** Option B (Cloudflare Tunnel) as the cheapest, safest near-term fix. Option D if we plan to formalise staging before launch.
