## Goal
Build the `/invite/:token` route and view in the tenant web app so new tenants can accept their invite, set a password, and reach the portal without agent intervention.

## Acceptance criteria
- [x] Route `/invite/:token` added to `tenant/src/router/index.ts`
- [x] GET: pre-fills email (read-only) from backend token lookup; prompts for full name + password
- [x] POST: calls `AcceptInviteView`, logs the user in on success, redirects to welcome/home screen
- [x] Expired or cancelled invite shows a clear error message with agent contact prompt
- [x] Invalid/unknown token returns a 404-style error screen (not a blank login page)
- [x] At least one integration test covering the happy path and the expired-token path

## Files likely touched
- `tenant/src/router/index.ts` — add `/invite/:token` route
- `tenant/src/views/auth/InviteAcceptView.vue` — new view (create)
- `backend/apps/accounts/views.py` — `UserInviteAcceptView` (GET + POST, line ~381)
- `content/emails/invite_tenant.md` — verify `invite_url` destination matches new route

## Test plan
**Manual:**
- Send a real invite to a test tenant email; click link; confirm InviteAcceptView loads with email pre-filled
- Fill name + password → submit → confirm redirect to home screen
- Use an expired token (modify DB expiry) → confirm error state
- Use a non-existent token → confirm 404 error state

**Automated:**

Uses same list-section/list-row CSS as LoginView. On success stores JWT via `auth._setTokens()` then redirects to `welcome` (or `2fa-enroll` if role requires TOTP).
- Added `/invite/:token` route (public, depth 0) to `tenant/src/router/index.ts`.
- Updated `backend/apps/accounts/admin_views.py` `_send_invite_email`: invite links now use `/invite/{token}` path (was `/accept-invite?token={token}` query-param form). Backend `AcceptInviteView` still uses `GET ?token=` query param internally — the frontend reads the path param and passes it as a query param on the GET and as body on the POST.
- Fixed a latent bug in `AcceptInviteView.get` and `.post`: both now catch `DjangoValidationError` in addition to `ValueError` — non-UUID strings (e.g. empty string or garbage) would previously blow up with an unhandled 500. In-scope sub-issue, fixed in this task.
- Added `backend/apps/accounts/tests/test_invite_accept.py` — 12 integration tests covering GET (valid, unknown, invalid-string, missing, cancelled, already-used) and POST (happy path ×2, already-accepted, unknown token, missing password, weak password). All 12 pass.
- Updated `content/emails/invite_tenant.md` with `invite_url_pattern` note documenting the new `/invite/{token}` URL scheme.

Blocking issues:

1. REGRESSION — agent/owner/supplier invites now 404. `backend/apps/accounts/admin_views.py:145` now emits `{base_url}/invite/{token}` unconditionally for ALL invite roles, but the admin app (`admin/src/router/index.ts:32`) only declares `/accept-invite`, not `/invite/:token`. Since `_resolve_base_url` is derived from the inviter's Origin (typically admin at `localhost:5173`), every agent/owner/supplier invite link will land on the admin app at an unregistered path. Previously `/accept-invite?token=...` matched admin. Fix options (pick one):
   - Branch `invite_url` by `invite.role` — tenant → `{tenant_base_url}/invite/{token}`, others → `{admin_base_url}/accept-invite?token={token}`. Requires a `TENANT_APP_BASE_URL` setting (and test).
   - Add a mirror route `/invite/:token` to `admin/src/router/index.ts` that renders `AcceptInviteView` (reading the path param) — cheaper, keeps one URL shape.
   Either way, add a test that asserts the generated URL for a non-tenant invite is reachable in the admin app's route table, or at minimum a backend unit test asserting the correct host/path per role.

2. Missing integration test for the path-param → query-param translation on the tenant side. `tenant/src/views/auth/InviteAcceptView.vue:193` reads `route.params.token` and passes it as `?token=` to the backend — this is the entire new contract and there's no frontend unit/E2E test asserting it. At least add a note for the tester to cover it in QA, or a Vitest component test. Not strictly blocking if the tester picks it up, but flag in handoff.

Non-blocking observations (do not need to fix for this ticket):
- `tenant/src/views/auth/InviteAcceptView.vue:205` — the `'already exists'` branch on GET is dead (only POST returns that detail). Harmless, but consider dropping to avoid future confusion.
- Rate limiting already present (`InviteAcceptThrottle` 5/min/IP) — adequate for brute force on GET; no POPIA issue with the email pre-fill since disclosure is bound to a random 128-bit UUID token.
- 6 states (loading/form/invalid/cancelled/used/success) are all reachable; the handoff note saying "5 states" is just a counting slip.
- 12 tests are substantive — they check status codes, JWT tokens, user creation, and invite.accepted_at mutation. Not tautological.

Send back for fix #1; fix #2 is optional but recommended.
