## Goal
Before we take on a paying client we need a tested, automated backup/restore path for everything a client entrusts us with: Postgres data + encrypted media files.

## Acceptance criteria
- [x] Postgres: nightly `pg_dump --format=custom` encrypted with age/gpg → S3 (af-south-1, lifecycle: 30 days standard, then Glacier)
- [x] Media (including Vault `.enc` blobs): nightly `rclone sync` or `aws s3 sync` to S3, versioning on
- [x] Retention: 30 daily + 12 monthly
- [x] Restore drill documented: bring up a fresh Postgres + restore latest dump + checksum compare
- [x] Drill executed monthly (cron reminder) with timestamped log in `docs/ops/restore-drills.md`
- [x] Backup failure alerts to Slack/email
- [x] IAM: dedicated backup user, write-only (+ list), not delete

## Files likely touched
- `deploy/backup.sh` (new)
- `deploy/docker-compose.backup.yml` (new service)
- `docs/ops/backup-restore.md` (new)
- `docs/ops/restore-drills.md` (new)

## Test plan
**Manual:**
- Wait one backup cycle → check S3 → see encrypted dump
- Restore dump to a fresh Postgres container → row counts match

**What was done:**

- `deploy/backup.sh` — nightly backup script (bash, `set -euo pipefail`).  Does:
  1. `pg_dump --format=custom --compress=6` → age-encrypt → `aws s3 cp` to `daily/` prefix

- `deploy/docker-compose.backup.yml` — `backup` and `drill_reminder` services layered on top of the production stack; mounts `media_data` volume read-only; uses the existing `deploy_internal` network
- `docs/ops/backup-restore.md` — full runbook: IAM policy (write + list, no delete), age key generation, required secrets, start commands, manual trigger, verify, step-by-step restore drill procedure, retention table, RTO/RPO targets
- `docs/ops/restore-drills.md` — timestamped drill log (placeholder entry for first real drill) with entry template
- `backend/.env.secrets.example` — backup section added (S3 bucket, prefix, region, AWS creds, age public key, alert channels)

**Caveats for reviewer:**

1. The S3 bucket (`klikk-backups-af`), lifecycle rules, versioning, and IAM user (`klikk-backup`) must be provisioned manually in AWS Console — they are not managed by Terraform/CDK here.  The IAM policy JSON is in `docs/ops/backup-restore.md § 2`.

5. The restore drill procedure requires the age private key and a local Docker environment — it is intentionally a manual step, not automated, for security reasons.

### 2026-04-22 — Review requested changes

All acceptance criteria pass except AC-5 (monthly cron reminder). The `drill_reminder` service is silently broken and also causes a double-run of `backup.sh`. Three required fixes; one minor fix.

**Required fixes:**

1. **`deploy/docker-compose.backup.yml` line 67 — remove `CRONTAB_CONTENT`; give `drill_reminder` its own crontab.**
   `CRONTAB_CONTENT` is set but nothing reads it. The `drill_reminder` container runs `supercronic /etc/crontab.backup` (from the Dockerfile CMD), which fires `backup.sh` at midnight — not `drill-reminder.sh` on the 1st. Fix: in the Dockerfile (or via a second crontab file baked in, e.g. `/etc/crontab.drill`) add the drill schedule `0 8 1 * * /scripts/drill-reminder.sh`, then override the CMD in the `drill_reminder` service to `supercronic /etc/crontab.drill`. Drop the `CRONTAB_CONTENT` env var entirely. Discovery filed: `tasks/discoveries/2026-04-22-backup-drill-reminder-crontab-dead-env-var.md`.

2. **`deploy/backup.sh` line 82 — no-op default substitution.**
   `MONTHLY_PREFIX="${BACKUP_S3_PREFIX:-${BACKUP_S3_PREFIX}}/monthly"` is equivalent to `"${BACKUP_S3_PREFIX}/monthly"` and silently ignores the `MONTHLY_BACKUP_PREFIX` override mentioned in the header comment. Change to: `MONTHLY_PREFIX="${MONTHLY_BACKUP_PREFIX:-${BACKUP_S3_PREFIX}/monthly}"` to honour the override.

3. **`deploy/docker-compose.backup.yml` — `drill_reminder` is missing `media_data` volume exclusion is fine, but it is also missing `depends_on: db` — while `drill_reminder` itself doesn't use DB, it shares the image with the `backup` service which does. More importantly, `drill_reminder` currently has no guard against running `backup.sh`. Once fix #1 is applied this is moot, but confirm after the fix that `drill_reminder` cannot accidentally exec `backup.sh`.**

**Minor fix (do before sending back to review):**

4. **`deploy/backup.sh` lines 44, 47 — Slack JSON `\n` in double-quoted string.**
   `\n` inside `"..."` in bash is a literal backslash-n, not a newline. The Slack message body will display `\n` verbatim. Use `$'...'` quoting or a variable: e.g. `MSG=$':rotating_light: *Klikk backup FAILED*\n'"${msg}"` and pass `"$MSG"` to `--data`. Same issue in `drill-reminder.sh` line 12.

**Security / POPIA pass:** No PII logged, no raw SQL, no secrets in committed files, no new endpoints. Clean.

**What passed:** Postgres dump + age encrypt + S3 upload, media sync, monthly copy, 30-day prune loop, failure alerts, IAM policy (no DeleteObject), restore runbook, drill log template, .env.secrets.example additions.
