## Goal
Fix the role-alias mismatch that causes `extract_signer_fields` to return 0 signing fields for a tenant signer, making it impossible to complete native signing for any template where the HTML uses `data-signer-role="tenant"`.

## Acceptance criteria
- [x] Either normalise template HTML roles to `tenant_1` / `tenant_2` during `_convert_legacy_signing_span`, OR extend `extract_signer_fields` to treat `"tenant"` as an alias for `"tenant_1"` (and `"landlord"` â†’ `"landlord_1"` etc.) â€” **both** applied
- [ ] `esigning_native_signing_test` Step 5 returns > 0 fields for any template that has a tenant signature block (manual E2E â€” for tester)
- [ ] Template 57 (and any other template with `data-signer-role="tenant"`) produces signing fields when a `tenant_1` signer creates a native submission (manual â€” for tester)
- [x] No regression in existing `tenant_1` / `landlord_1` explicit-role templates â€” covered by `test_no_cross_role_contamination` + `test_explicit_tenant_1_role_matched`
- [x] Regression test added: `test_extract_signer_fields_tenant_alias` â€” asserts fields are found when signer record role is `tenant_1` but HTML uses `data-signer-role="tenant"`

**Blocker: asymmetric alias maps cause a landlord-path regression.**

`_role_to_tiptap` (services.py:571-577, used by `create_native_submission` to populate `signer_records[].role`) maps:
- `landlord â†’ landlord` (no `_1` suffix)
- `lessor â†’ landlord`
- `tenant â†’ tenant_1`

The new `_legacy_role_alias` (services.py:450-456) and `_html_role_aliases` (services.py:618-624) maps, however, both canonicalise:
- `landlord â†’ landlord_1`
- `lessor â†’ landlord_1`

This means:

1. For any legacy template (`<span data-type="signature-block" data-signer-role="landlord">`), `_convert_legacy_signing_span` now emits HTML with `role="landlord_1"`. But the signer record still stores `role="landlord"` (via `_role_to_tiptap`). `extract_signer_fields(html, "landlord")` then compares HTML canonical `"landlord_1"` against signer_role `"landlord"` â†’ 0 fields. **This is a new regression introduced by the fix**: every landlord-signing flow against a legacy template that was working before this patch now breaks.

2. Confirmed by searching the codebase: `landlord_1` exists only in this PR + its tests; `SignatureBlockNode.ts` (admin TipTap) uses `'landlord'` (line 79, 84, 89, 94, 107). The canonical landlord tiptap role in the rest of the system is unsuffixed.

**Required fixes (choose one internally consistent approach):**

1. Align the maps. Either:
   - **Option A (recommended, smaller blast radius):** change both `_legacy_role_alias` and `_html_role_aliases` so `landlord â†’ landlord` and `lessor â†’ landlord` (keep them unsuffixed to match `_role_to_tiptap` + `SignatureBlockNode.ts`). The entries become no-ops for `landlord` but `lessor â†’ landlord` still adds value.
   - **Option B:** also change `_role_to_tiptap` to `landlord â†’ landlord_1` / `lessor â†’ landlord_1` and migrate any stored signer records. This is a much bigger change â€” do not pick this without PM sign-off.

2. Deduplicate the two alias maps. They encode the same concept and must stay in lock-step â€” extract a single module-level constant (e.g. `_HTML_ROLE_CANONICAL: dict[str, str]`) and reference it from both `_convert_legacy_signing_span` and `extract_signer_fields`. As written, the next person to add a role (`witness_1`? `agent`?) will update one and forget the other.

3. Add a regression test for the landlord path mirroring `test_extract_signer_fields_tenant_alias` â€” i.e. signer record role is whatever `_role_to_tiptap` produces for `"Landlord"` input, HTML role is `"landlord"` legacy â€” fields must be found. The current `test_landlord_alias_matched` tests `landlord_1` as the signer_role, which is **not what the production caller passes**; this test passes the unit but doesn't prove the end-to-end path works. Fix the test to exercise the actual `create_native_submission` â†’ `extract_signer_fields` contract.

4. Minor: `owner` (rental-mandate signing per `_role_to_tiptap`) is absent from both alias maps. Confirm no legacy mandate templates use `data-signer-role="owner"`; if any do, add a passthrough entry or a comment explaining why none is needed.
