## Goal

- [x] `website/src/pages/legal/privacy.astro` — POPIA-compliant Privacy Policy, SA context, lists info officer contact, lawful bases (s11), retention, DSAR process, cross-border (s72 — we are af-south-1 only)

- [x] `website/src/pages/legal/popia.astro` — POPIA statement + Operator/Responsible Party framing
- [x] `website/src/pages/legal/paia-manual.astro` — PAIA s51 manual with Form 2 template download

- [x] Reviewed against `klikk-legal-POPIA-RHA` skill; defers to that skill for SA citations

### 2026-04-22 — reviewer (changes requested)

**Review requested changes:**

1. **`backend/apps/legal/views.py:57-62` — `POST /consent/` always returns HTTP 200, never 201.**
   The `created` variable is computed but never used — `return Response(..., status=status.HTTP_200_OK)` is hardcoded on every path. Fix: capture the second return value of `get_or_create` in the serializer's `create()` method, bubble it up (e.g. return a tuple or set an attribute on the instance), and in the view return `HTTP_201_CREATED` when a new row was created and `HTTP_200_OK` when idempotent. Also remove the dead `created = consent.accepted_at is not None` line — `accepted_at` is `auto_now_add=True` so this is always `True`.

2. **`admin/src/views/auth/RegisterView.vue` + registration API — POPIA s11 consent not persisted to `UserConsent`.**
   `tosAccepted` is a front-end-only gate (button disable). `handleRegister()` does not pass `tos_accepted` to the backend, and the register endpoint does not call `POST /api/v1/legal/consent/` after account creation. This means consent is never recorded in `UserConsent`, defeating the purpose of the model. Fix (choose one):
   - (A) After the `auth.register(...)` call succeeds, fetch the current ToS document ID from `/api/v1/legal/documents/current/` and POST to `/api/v1/legal/consent/` with the new user's token before redirecting; or
   - (B) Have the Django register endpoint accept a `tos_document_id` field, validate it, and create the `UserConsent` row server-side before returning the JWT.
   Option B is preferred for POPIA audit integrity (server records the IP + UA at the moment of registration, not a subsequent API call).

3. **`backend/config/urls.py:8,19` — out-of-scope `contact_view` wiring.**
   The import `from .contact import contact_view` and `path("api/v1/contact/", contact_view, ...)` are unrelated to OPS-004. `backend/config/contact.py` already existed but was un-wired. Either confirm this was intentional and add it to a separate task, or revert these two lines from this commit. Not blocking on its own, but please confirm intent.

**Discovery filed:** `tasks/discoveries/2026-04-22-xff-trust-all-apps.md` — shared XFF spoofing risk across `apps/legal`, `apps/esigning`, `apps/accounts`; out of scope for this task.

**What was done well:** Models are clean, UniqueConstraints correct, migration matches models, admin registration is thorough, test coverage is present (unit + integration), all four Astro pages built cleanly, POPIA citations are correct, Information Officer placeholder is clearly called out.

### 2026-04-22 — implementer (re-work, reviewer issues 1–3)

**Issue 1 — `POST /consent/` always returned 200 (fixed):**
- `backend/apps/legal/serializers.py`: `UserConsentSerializer.create()` now captures the second value from `get_or_create` as `created` and sets `consent._created = created`. The dead `consent.accepted_at is not None` line is removed.
- `backend/apps/legal/views.py`: `UserConsentView.post()` reads `getattr(consent, "_created", False)` and returns `HTTP_201_CREATED` on a new row, `HTTP_200_OK` on an idempotent duplicate.

**Issue 2 — POPIA s11 consent not persisted on registration (fixed, Option B):**
- `backend/apps/accounts/serializers.py`: `RegisterSerializer` gains two optional write-only fields `tos_document_id` and `privacy_document_id`. `create()` pops them from `validated_data`, then calls the new `_record_consent()` static method which resolves each ID against `LegalDocument.objects.get(pk=..., is_current=True)` and calls `UserConsent.objects.get_or_create()` with IP + user-agent from `request.META`. Stale/invalid IDs are silently skipped so registration is never hard-blocked by a missing document row.
- `backend/apps/accounts/views.py`: `RegisterView.post()` now passes `context={"request": request}` to `RegisterSerializer` so the serializer can extract IP/UA.
- `admin/src/stores/auth.ts`: `register()` payload type extended with optional `tos_document_id` and `privacy_document_id` fields.
- `admin/src/views/auth/RegisterView.vue`: On `onMounted`, `GET /api/v1/legal/documents/current/` is called (non-blocking; errors silently skipped) to fetch the current ToS and Privacy Policy IDs. These are stored in `tosDocumentId` and `privacyDocumentId` refs and passed to `auth.register()` in `handleRegister()`.
- Note: `admin/src/api.ts` and `admin/src/stores/auth.ts` were found deleted from the working tree (unrelated to this task). Both were restored from HEAD before applying changes.

**Issue 3 — out-of-scope `contact_view` in `urls.py` (reverted):**
- `backend/config/urls.py`: Removed the `from .contact import contact_view` import and `path("api/v1/contact/", contact_view, ...)` route. The `contact.py` module remains intact; wiring it is a separate task.
