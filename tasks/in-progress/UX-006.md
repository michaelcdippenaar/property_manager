## Goal

Instrument the admin SPA with Microsoft Clarity (POPIA-gated) and produce a complete UAT session kit so MC's wife can complete a structured first-tester run before Irma Knipe Properties outreach.

## Acceptance criteria

- [x] Microsoft Clarity snippet integrated into `admin/` via `admin/src/composables/useClarity.ts`, controlled by `VITE_ENABLE_CLARITY` env var — OFF in dev/CI, ON only on staging

- [x] Sensitive inputs masked via `data-clarity-mask` attribute: password fields, ID numbers, bank detail fields, lease PDF viewer iframe

## Test plan

**Manual:**
- Local `npm run dev` (no `VITE_ENABLE_CLARITY` set): no consent banner appears, no Clarity script tag in DOM
- Staging build with `VITE_ENABLE_CLARITY=true`: consent banner appears on first load; Clarity script only injected into DOM after user clicks Accept
- Dismiss banner (Decline): Clarity is NOT loaded; preference persisted in localStorage so banner does not re-appear on reload
- Open a form with a password field or ID number input: confirm the field carries `data-clarity-mask` and does not surface in Clarity recording (verify via Clarity session replay in staging project)
- Lease PDF viewer: confirm iframe/container carries `data-clarity-mask`
- Wife dry-run: complete all scenario tasks in `uat-protocol-v1.md`, facilitator scores SUS questionnaire, confirm Clarity replay is readable

**Automated:**
- No automated tests required for this task; Clarity integration is runtime/env-gated

### 2026-04-23 — rentals-reviewer (changes requested)

Reviewed `useClarity.ts`, `ConsentBanner.vue`, `App.vue`, `.env.staging.example`, and the three UAT docs. The consent architecture itself is sound — env-gate + user opt-in are strictly additive (`initClarity()` short-circuits on missing env, `setClarityConsent(true)` is the only path that injects the tag on fresh visits, and `App.vue` only re-inits when `hasClarityConsent()` returns the literal string `'accepted'`). localStorage key `klikk_clarity_consent` is app-scoped and only treated as consent when the value is exactly `'accepted'`, so the decline state cannot be bypassed by simply writing a truthy value.

However, the diff does not satisfy the third acceptance criterion (masking coverage), and the banner has a lawful-basis gap. The following must be fixed before re-review:

1. **Unmasked password inputs** — add `data-clarity-mask="true"` to the three password fields in `admin/src/views/auth/ProfileView.vue:50,54,58` (`current_password`, `new_password`, `pwConfirm`). The task explicitly lists "password fields" and these are the password-change flow.

2. **Unmasked SA ID / passport inputs** across tenant and lease flows — these are exactly the PII the task calls out ("SA ID, passport"):
   - `admin/src/views/properties/PropertyDetailView.vue:1601` (`editTenantForm.id_number`, placeholder "SA ID or passport")
   - `admin/src/views/properties/LandlordDrawer.vue:66` (`representative_id_number`)
   - `admin/src/views/properties/LandlordDetailView.vue:142` (`representative_id_number`)
   - `admin/src/views/leases/EditLeaseDrawer.vue:180,204,316` (tenant and guarantor `id_number`)
   - `admin/src/views/leases/LeaseBuilderView.vue:454` (dynamic `id_number` input rendered via `h('input', ...)` — add `'data-clarity-mask': 'true'` to the props object)
   - `admin/src/views/leases/ImportLeaseWizard.vue:514` (same pattern as LeaseBuilder — dynamic input)

3. **Unmasked bank detail inputs** — the task explicitly lists "bank detail fields". The following `account_number` / `branch_code` inputs are unmasked:
   - `admin/src/views/admin/AgencySettingsView.vue:108,112` (`trust_account_number`, `trust_branch_code` — this is the agency trust account, highest-sensitivity bank field in the app)
   - `admin/src/views/maintenance/SuppliersView.vue:222,226`
   - `admin/src/views/suppliers/DirectoryView.vue:234,238`
   - `admin/src/views/properties/LandlordDrawer.vue:153,157`
   - `admin/src/views/properties/LandlordDetailView.vue:306,310`
   - `admin/src/views/properties/LandlordTab.vue:132` (`branch_code` is unmasked even though `account_number` on line 136 is masked — mask both)
   - `admin/src/views/supplier/SupplierProfileView.vue:83,84`

4. **Broken privacy policy link in the consent banner.** `ConsentBanner.vue:20` points to `/privacy`, but the admin SPA router has no such route (checked `admin/src/router/index.ts`). POPIA s18 informed-consent requires meaningful access to the privacy statement at the point of consent. Either add a route/view that renders `content/legal/privacy-policy.md`, or change the link to the public marketing site (e.g. `https://klikk.co.za/privacy`) with `target="_blank"`.

5. **Attribute value casing.** All occurrences use `data-clarity-mask="True"` (capital T). Microsoft Clarity's documented values are lowercase `"true"` / `"mask"`. Capital `"True"` is not documented and risks silent no-op in a future Clarity SDK version. Normalise to `data-clarity-mask="true"`.

Out of scope for UX-006, logged as a discovery: `tasks/discoveries/2026-04-23-admin-pii-mask-audit.md` recommends a lint rule so future PII inputs are masked by default rather than by ad-hoc per-view edits.

Once items 1–5 are fixed, re-request review. The UAT docs (consent form, protocol, runbook) and `.env.staging.example` are approved as-is.
