### 2026-04-22 — rentals-reviewer

**Review requested changes**

Three issues must be fixed before this passes.

**1. `website/src/components/Header.astro` lines 12–17 — `role="listitem"` on `<a>` elements (blocking)**

`<a href="..." role="listitem">` is incorrect ARIA. Applying `role="listitem"` to an `<a>` element overrides its implicit `link` role. Screen readers will announce these items as "listitem" rather than as navigable links, breaking keyboard-and-SR navigation in the main nav. WCAG 4.1.2 requires elements to have correct name, role, and value.

Fix: Remove `role="list"` from the `<div class="nav-links">` and remove `role="listitem"` from all five `<a>` elements. The `<nav aria-label="Main navigation">` wrapper already establishes the landmark — the links do not need an additional list role. Plain `<a>` elements inside `<nav>` is the correct, accessible pattern.

**2. `admin/src/views/auth/AcceptInviteView.vue` and `admin/src/views/auth/ResetPasswordView.vue` — F-004/F-005 violations not addressed (blocking)**

The acceptance criterion states "all auth form label/input pairs now use `for`/`id`". Two auth views were missed:

- `AcceptInviteView.vue` lines 48, 54, 59, 66, 89: All five `<label>` elements have no `for` attribute; corresponding `<input>` elements have no `id`.
- `AcceptInviteView.vue` line 100: Error div has no `role="alert"`.
- `ResetPasswordView.vue` lines 13, 17: Both `<label>` elements have no `for` attribute.
- `ResetPasswordView.vue` line 20: Error div has no `role="alert"`.

Apply the same fix pattern used in `ForgotPasswordView.vue`: add matching `for`/`id` pairs, `role="alert"` on error divs, `aria-hidden="true"` on alert icons, and relevant `autocomplete` attributes.
