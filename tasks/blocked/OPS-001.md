## Goal
Stand up a CI/CD pipeline so every PR is linted/tested automatically and every main-branch merge deploys to staging. Without this, regressions land silently and deploys are manual + error-prone.

## Acceptance criteria
- [x] `.github/workflows/ci.yml`: runs on PR — black/ruff, eslint, `pytest backend`, `npm test` for admin + web_app, `flutter analyze`
- [x] `.github/workflows/staging-deploy.yml`: on merge to main → build Docker images, push to registry, SSH + `docker compose pull && up -d` on staging host
- [ ] Branch protection: PR requires green CI to merge
- [x] Status badge in repo README
- [x] Secrets managed as GitHub Actions secrets (SSH key, registry creds, env)
- [x] Rollback: previous image tag kept, documented one-command rollback
- [x] Staging deploy workflow uses `workflow_dispatch` (manual trigger) — **no auto-deploy on push to main until MC has manually tested the staging environment**. A branch-protected approval gate is an acceptable alternative. This is a hard requirement from MC (DEC-015 answered 2026-04-22).

## Staging server details (DEC-015 answered 2026-04-22)
- Staging URL: https://www.klikk.co.za
- IP: 102.135.240.222 (public — no tunnel required)
- `STAGING_HOST` secret should be set to `102.135.240.222` (or `www.klikk.co.za`)
- **MC must manually approve / trigger all staging deploys. Do not set up auto-deploy on push to main.**

### 2026-04-22 (re-implementation) — implementer

**Fix 1 — DB env vars (blocker):** Replaced the single `DATABASE_URL` env var in the `Run pytest` step of `.github/workflows/ci.yml` with the six individual decouple vars (`DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT`) that `backend/config/settings/base.py` actually reads. The service container still creates the database as `klikk_test`; the env vars now match, so Django's `DATABASES["default"]` will resolve correctly.

**Fix 2 — GHCR_PAT heredoc injection (security):** Replaced the raw `ssh … << EOF` block in `.github/workflows/staging-deploy.yml` with `appleboy/ssh-action@v1`. The PAT is now passed via `envs: IMAGE_TAG,GHCR_OWNER,GHCR_PAT` and the `env:` block on the step — appleboy forwards these as SSH environment variables to the remote session. The script uses `"$GHCR_PAT"` (shell variable, not `${{ }}` Actions expression), so the secret value is never embedded as a literal string in the heredoc payload. The now-redundant `actions/checkout` and manual `Write SSH key` steps in the deploy job were removed — appleboy handles SSH key setup internally via its `key:` input.
