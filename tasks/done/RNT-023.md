## Goal
Ensure that navigating to `/leases?expand=124&sign=1` reliably opens and holds the signing modal for lease 124, without flashing and self-dismissing.

## Repro
- URL: `http://localhost:5173/leases?expand=124&sign=1`
- Expected: leases list opens, lease 124 expands, signing modal auto-launches and persists.
- Actual: page flashes — modal appears for a single frame then dismisses. No backend 500 (backend log silent).

## Additional repro (2026-04-25)
Navigating to `http://localhost:5173/leases` and clicking the "All" tab/filter also causes the page to flash. The flash bug is broader than just the URL-driven sign flow — it triggers on the All-filter click too. Same likely root cause family: route/query-param watcher loop, store action triggering `router.replace`, or the LeasesView re-mounting on filter change.

## Root cause candidates
1. `?sign=1` is consumed on mount but a route guard or watcher re-navigates and strips the param, unmounting the modal.
2. Modal opens before lease data has loaded (`leases[124]` is undefined); the open-handler bails out and closes itself.
3. `expand=124` watcher and `sign=1` watcher conflict — one stomps the other.
4. Pinia store action triggers a `router.replace` that removes `sign=1`.

### 2026-04-25 — implementer

**Root cause (confirmed):** Two separate bugs sharing the same family:

1. **All-tab flash / background-refresh flash** — `LeasesView` used `v-if="loading"` on the lease list container. Every call to `loadLeases()` sets `loading = true`, destroying the entire lease list DOM (including `ESigningPanel` instances and their open modals) for a frame. Fixed by introducing `initialLoad = ref(leases.length === 0)` and gating the skeleton to `v-if="initialLoad && loading"`. Background refreshes no longer destroy the DOM.

2. **Sign modal flash-and-dismiss** — `ESigningPanel` received an `:auto-open` prop evaluated from reactive `route.query`. The prop was re-evaluated before data loaded and while `?sign=1` was still being stripped, causing a race. Fixed by removing the `autoOpen` prop entirely. `LeasesView.initView()` now captures `expand`/`sign` synchronously before any awaits, then after data loads calls `panel.openModalWhenReady()` via template ref, then strips `?sign=1` via `router.replace({ query: restQuery })`.

**`ESigningPanel.openModalWhenReady()`** — new async method with a polling loop (50 ms interval, 5 s deadline) that waits for `loading.value` to clear then opens the modal only if no existing submissions are present. A `_mounted` flag guards against acting after unmount.

**Tests:** `admin/src/views/leases/__tests__/leases-flash.browser.test.ts` — 10 tests across 4 suites. All 10 pass. Because `LeasesView` uses `onActivated` (not `onMounted`), `defineExpose({ initView })` was added to `LeasesView.vue` so the test harness can call `initView()` directly on the component instance. The stub for `ESigningPanel` returns `openModal` and `openModalWhenReady` from `setup()` so they are reachable via the template ref map in `initView`.

**Manual verification:** Reviewer should confirm in dev browser that `/leases?expand=<id>&sign=1` opens the modal and persists, and that filter tab clicks no longer cause visible flashes.
