## Question
Confirm the production subdomain plan.

## Answer (FINAL 2026-04-22)

Authoritative subdomain map:

| Subdomain | Role |
|---|---|
| `klikk.co.za` | Marketing site (Astro) |
| `app.klikk.co.za` | Admin SPA (Vue 3) — **canonical** agent URL |
| `agent.klikk.co.za` | 301 → `app.klikk.co.za` |
| `tenant.klikk.co.za` | Tenant web app (`web_app/`) — **canonical** tenant URL |
| `mobile-agent.klikk.co.za` | **Quasar agent PWA** |
| `mobile-tenant.klikk.co.za` | **Quasar tenant PWA** (replaces Flutter tenant in v1) |
| `api.klikk.co.za` | Django API — **canonical** |
| `backend.klikk.co.za` | 301 → `api.klikk.co.za` (sunset after 90 days) |
| `status.klikk.co.za` | Uptime status page |
| `mail.klikk.co.za` | AWS SES transactional sender (isolated SPF/DKIM/DMARC) |

Google Workspace keeps its SPF/DKIM on the apex (`klikk.co.za`) so `mc@klikk.co.za` etc. are unaffected by SES auth on `mail.klikk.co.za`.

**v1 architecture change this implies:** tenant mobile surface is a Quasar PWA, not a Flutter app. One mobile stack for both agent + tenant.
