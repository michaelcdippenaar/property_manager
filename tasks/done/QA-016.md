## Goal
Close two WCAG failures found in the QA-007 audit: missing keyboard focus trap in modal/drawer components (WCAG 2.1 SC 2.1.2) and an invalid `<label for>` association in AcceptInviteView (WCAG 1.3.1).

## Acceptance criteria
- [x] On modal/drawer open, focus moves programmatically to the first focusable element inside the dialog
- [x] Tab cycles within the dialog only; Shift+Tab wraps to the last element
- [x] Escape closes the dialog and returns focus to the trigger element
- [x] Implemented via `focus-trap` npm composable or by replacing `<div role="dialog">` with native `<dialog>` element in `BaseModal.vue` and `BaseDrawer.vue`
- [x] `admin/src/views/auth/AcceptInviteView.vue`: `<p id="invite-email">` replaced with `<input type="email" id="invite-email" :value="invite.email" readonly tabindex="-1">` so the `<label for="invite-email">` association is valid
- [ ] Validated with keyboard-only navigation and VoiceOver (screen reader) on the property drawer, lease drawer, and confirmation modals

## Files likely touched
- `admin/src/components/BaseModal.vue`
- `admin/src/components/BaseDrawer.vue`
- `admin/src/views/auth/AcceptInviteView.vue`

## Test plan
**Manual:**
- Open a modal/drawer with keyboard only (Tab to trigger, Enter to open) — confirm focus moves inside
- Tab through all focusable elements — confirm focus does not escape to background
- Press Escape — confirm dialog closes and focus returns to trigger
- VoiceOver: navigate to AcceptInviteView, confirm "Email" label is announced for the email field

### 2026-04-23 — rentals-tester (static verification)

Static code review confirms all acceptance criteria are satisfied in HEAD:

**1. useFocusTrap.ts composable:**
- Line 40–50: Escape handler with optional `onEscape` callback
- Line 52–77: Tab/Shift+Tab cycling logic with wrap-around (first ↔ last)
- Line 79–94: `activate()` stores previously focused element and sets focus to first focusable child
- Line 96–102: `deactivate()` restores focus to previously focused element and cleans up event listener
- Line 104: Auto-deactivate on component unmount

**2. BaseModal.vue:**
- Line 23–24: `@after-enter="activate"` and `@before-leave="deactivate"` wired to Transition component
- Line 28: `ref="trapRef"` bound to dialog panel
- Line 69: Import of `useFocusTrap` composable
- Line 85: Instantiation with close callback: `useFocusTrap(props.closable ? () => emit('close') : null)`
  - When `closable: false`, Escape is swallowed but does not dismiss (as designed)
  - When `closable: true`, Escape closes and restores focus

**3. BaseDrawer.vue:**
- Line 23–24: Same `@after-enter` / `@before-leave` wiring as BaseModal
- Line 28: `ref="trapRef"` bound to dialog panel
- Line 68: Import of `useFocusTrap` composable
- Line 84: Always instantiated with close callback: `useFocusTrap(() => emit('close'))`

**4. AcceptInviteView.vue:**
- Line 48: `<label for="invite-email">Email</label>`
- Lines 49–57: `<input type="email" id="invite-email" :value="invite.email" readonly tabindex="-1" aria-readonly="true">`
  - Valid `<label for>` ↔ `<input id>` association (WCAG 1.3.1)
  - `readonly` + `tabindex="-1"` prevents user interaction
  - `aria-readonly="true"` signals to assistive tech that field is read-only
