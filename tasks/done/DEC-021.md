## Question

In what order do we deploy to staging vs production? Does every OPS task need to target staging first, or can implementers go straight to production subdomains?

## Options (summarised)

- **Staging-first (recommended):** Each service is deployed and verified on a `staging-*` subdomain before production DNS is cut over. Smoke tests, RBAC checks, and E2E suites must pass on staging before the matching production subdomain goes live.
- **Parallel:** Staging and production deployed simultaneously; rollback if prod breaks.
- **Production-only:** Deploy straight to production; use feature flags to limit exposure.

## Answer

FINAL (2026-04-24 — MC):

**Staging-first. No surface goes live on a production subdomain until staging smoke + RBAC + E2E pass.**

### Subdomain convention

| Staging subdomain | Production subdomain | Surface |
|---|---|---|
| `staging-www.klikk.co.za` | `klikk.co.za` | Marketing site (Astro) |
| `staging-app.klikk.co.za` | `app.klikk.co.za` | Admin SPA |
| `staging-mobile-agent.klikk.co.za` | `mobile-agent.klikk.co.za` | Agent app (Quasar/Capacitor) |
| `staging-tenant.klikk.co.za` | `tenant.klikk.co.za` | Tenant web app |
| `staging-api.klikk.co.za` | `api.klikk.co.za` | Django API backend |

### Sequencing rule

1. Implementer deploys to staging subdomain via `workflow_dispatch` (OPS-001 CI/CD).
2. Tester runs smoke test and relevant E2E suite against the staging subdomain.
3. On green: MC (or designated ops) cuts over production DNS — the production subdomain goes live.
4. Any regression on staging → blocked; production subdomain not created/pointed.

### Per-task implications

- **OPS-001 (CI/CD):** `staging-deploy.yml` targets staging subdomains. A separate `prod-deploy.yml` (future) is gated on staging green.
- **OPS-006 (DNS/TLS/email):** Phase 1 = staging subdomains + certs. Phase 2 = production subdomains, only after staging verified.
- **OPS-002 (Sentry):** Three separate Sentry environments: `dev`, `staging`, `production`. DSNs are distinct per environment.
- **OPS-008 (uptime monitoring):** Staging endpoints monitored at 10-min intervals, no SLA. Production endpoints at 5-min intervals, full SLA alerting.
- **OPS-009 (secrets vault):** SSM parameter paths are `/klikk/staging/*` for staging and `/klikk/prod/*` for production — separate IAM policies.
- **RNT-SEC-001 (secret rotation):** Rotated secrets written to `/klikk/staging/*` first; verified on staging; then written to `/klikk/prod/*`.
