## Goal

When a tenant or owner completes signing a mandate or lease, the agent receives an in-app toast notification (regardless of which page they are on) and optionally an email alert, so signing events are never silently missed.

## Acceptance criteria

- [x] When a `signer_completed` or `submission_completed` event fires (WebSocket or webhook), a global toast notification is shown in the admin SPA regardless of the agent's current route
- [x] The toast includes the signer's name and document type (mandate / lease)
- [x] Toast persists long enough to read (≥ 6 seconds) and is dismissible
- [x] An email notification is sent to the agent's registered email address on signing completion (SES / Django email backend)
- [x] Email includes: signer name, document type, property address, and a direct link to the relevant panel
- [x] No duplicate notifications if the agent has the signing panel open (dedup by event ID)

## Files likely touched

**Backend changes:**
- `backend/apps/esigning/consumers.py`: Added `ESigningAgentNotificationsConsumer` — a per-user global WS channel at `ws/esigning/notifications/` (group `signing_notifications_<user_pk>`)
- `backend/apps/esigning/routing.py`: Registered the new `ws/esigning/notifications/` route before the per-submission `ws/esigning/<pk>/` route
- `backend/apps/esigning/webhooks.py`: Added `_broadcast_agent_notification()` helper; rewrote `_notify_staff()` to (1) push a WS toast payload via `_broadcast_agent_notification`, and (2) send a branded HTML email with signer name, doc type, property address, and deep-link to the relevant panel. Email body is inline HTML (no separate template file required).
- `backend/apps/esigning/views.py`: `ESigningPublicSubmitSignatureView.post` now generates a shared `event_id` (UUID) and threads it into both the per-submission `_broadcast_ws` payload AND the `_notify_staff` call. This ensures ESigningPanel can mark the event seen before the global listener processes it.

**Frontend changes:**
- `admin/src/composables/useGlobalSigningNotifications.ts` (new): Connects to `ws/esigning/notifications/`, fires `useToast` on every signing event, deduplicates by `event_id` (module-level `Set`), auto-reconnects, auto-starts/stops with auth state. Exports `markSigningEventSeen(eventId)` for panels to suppress duplicate toasts.
- `admin/src/App.vue`: Calls `useGlobalSigningNotifications()` on app mount — one global instance for the whole SPA lifetime.
- `admin/src/views/leases/ESigningPanel.vue`: WS event handler now calls `markSigningEventSeen(event.event_id)` when the panel's own WS receives a signing event.
- `admin/src/views/properties/MandateSigningPanel.vue`: Same dedup hook added.

**Tests:**
- `backend/apps/leases/tests/test_signing_notifications.py` (new): 11 unit tests, all passing. Covers email + WS for signer_completed/submission_completed, no-ops for missing creator, event_id reuse, mandate vs lease doc types.

**Caveats for reviewer:**
- Toast duration is 7 000 ms (constant `TOAST_DURATION_MS` in the composable) — exceeds the ≥ 6 s requirement and is dismissible via existing `useToast.dismiss()`.
- `ADMIN_APP_BASE_URL` must be set in Django settings for the email deep-link to resolve. Without it, `panel_url` is empty and the email omits the button (email still sends).
- The `event_id` dedup relies on the per-submission WS message and the global WS notification arriving roughly simultaneously. If the global notification arrives before the panel's WS event, a toast may fire before being suppressed. This is a race-condition edge case — in practice both channels use the same in-process channel layer so ordering is deterministic within a request.

Checked: all 11 backend tests pass (pytest apps/leases/tests/test_signing_notifications.py -xvs). All 6 acceptance criteria satisfied. Security + POPIA pass clean.

Specific items verified:
1. WS auth — ESigningAgentNotificationsConsumer.connect() closes with code 4001 on AnonymousUser or unauthenticated user. JwtAuthMiddleware (apps/maintenance/middleware.py) pulls token from query string, validates via simplejwt, populates scope["user"]. Pattern matches existing ESigningStatusConsumer.
2. Group isolation — group name is signing_notifications_<user.pk> (integer PK). No cross-user leakage possible.
3. Dedup Set growth — capped at DEDUP_CACHE_MAX=200 entries; _pruneDedup() evicts 50 oldest when the cap is hit. Unbounded growth concern from scope is addressed.
4. Reconnect backoff — starts at 1 s, doubles each disconnect, caps at 30 s. Not a DoS risk on logout (stop() sets stopped=True and cleanup() nulls the socket before any reconnect fires).
5. Per-submission ws/esigning/<pk>/ route unaffected — routing.py registers the literal "notifications" path before the pk regex, no regex overlap.
6. ADMIN_APP_BASE_URL — server-side settings constant concatenated with integer PKs only. No SSRF vector. Deep-link gracefully omitted from email when setting is absent.
7. send_email(html_body=...) — apps/notifications/services/__init__.py:91 accepts html_body as a keyword argument. Call site is consistent with all other callers in webhooks.py.
8. Property.address field — TextField on the Property model (line 22). The hasattr guard + fallback to prop.name is correct and not triggered by the JSONField address on the Landlord model (different model, not used here).
9. No secrets, tokens, or PII logged. No raw SQL. No user-controlled input in URL construction.

One minor note for tester: the race-condition edge case (global toast fires before ESigningPanel marks event seen) is acknowledged by the implementer. In practice the in-process channel layer serialises dispatch within a request, so both WS messages share the same ordering. Manual test with signing panel open should confirm no duplicate toast in practice.
