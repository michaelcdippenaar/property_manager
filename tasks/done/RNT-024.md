2026-04-25 — rentals-implementer: Root cause confirmed — `badges()` action in `MaintenanceRequestViewSet` used a global, unscoped `MaintenanceRequest.objects.exclude(...)` count completely bypassing the role-based queryset scoping in `get_queryset()`. An agent user whose portfolio had no requests would see badge=5 (global total) while the list showed 0.

Fix applied in `backend/apps/maintenance/views.py` — `badges()` now calls `self.get_queryset()` (reusing the same scope logic: tenant sees own, non-admin agents see accessible properties, admin sees all) then filters to `status__in=[OPEN, IN_PROGRESS]`. The `pending_questions` count was also unscoped; it now applies the same `get_accessible_property_ids` scoping for non-admin users, with tenants returning 0 (they are not agents).

Full docstring added to `badges()` explaining the counting rule, scoping rationale, and status semantics.

New test file `backend/apps/maintenance/tests/test_badges.py` — 8 tests, all passing:
- Admin badge matches global open+in_progress count
- Agent badge isolates own properties (no cross-agent bleed)
- badge count == filtered list count for same user auth
- Closed/resolved requests excluded
- Mixed status counts correct
- AgentQuestion scoping: admin sees all, tenant sees 0, answered not counted

No frontend changes needed — the API contract (`open_issues`, `pending_questions` keys) is unchanged; only the values are now correct.
