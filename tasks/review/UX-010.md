## Goal
Build the `/invite/:token` route and view in the tenant web app so new tenants can accept their invite, set a password, and reach the portal without agent intervention.

## Acceptance criteria
- [x] Route `/invite/:token` added to `tenant/src/router/index.ts`
- [x] GET: pre-fills email (read-only) from backend token lookup; prompts for full name + password
- [x] POST: calls `AcceptInviteView`, logs the user in on success, redirects to welcome/home screen
- [x] Expired or cancelled invite shows a clear error message with agent contact prompt
- [x] Invalid/unknown token returns a 404-style error screen (not a blank login page)
- [x] At least one integration test covering the happy path and the expired-token path

## Files likely touched
- `tenant/src/router/index.ts` — add `/invite/:token` route
- `tenant/src/views/auth/InviteAcceptView.vue` — new view (create)
- `backend/apps/accounts/views.py` — `UserInviteAcceptView` (GET + POST, line ~381)
- `content/emails/invite_tenant.md` — verify `invite_url` destination matches new route

## Test plan
**Manual:**
- Send a real invite to a test tenant email; click link; confirm InviteAcceptView loads with email pre-filled
- Fill name + password → submit → confirm redirect to home screen
- Use an expired token (modify DB expiry) → confirm error state
- Use a non-existent token → confirm 404 error state

**Automated:**
- `cd backend && pytest apps/accounts/tests/test_invite_accept.py`
- E2E: `mcp__tremly-e2e__tenant_invite_accept_flow`

2026-04-23 implementer: Implemented all acceptance criteria.
- Created `tenant/src/views/auth/InviteAcceptView.vue` — public view with 5 states: loading, form, invalid (404-style), cancelled (410 with agent name), used (already-accepted), success. Uses same list-section/list-row CSS as LoginView. On success stores JWT via `auth._setTokens()` then redirects to `welcome` (or `2fa-enroll` if role requires TOTP).
- Added `/invite/:token` route (public, depth 0) to `tenant/src/router/index.ts`.
- Updated `backend/apps/accounts/admin_views.py` `_send_invite_email`: invite links now use `/invite/{token}` path (was `/accept-invite?token={token}` query-param form). Backend `AcceptInviteView` still uses `GET ?token=` query param internally — the frontend reads the path param and passes it as a query param on the GET and as body on the POST.
- Fixed a latent bug in `AcceptInviteView.get` and `.post`: both now catch `DjangoValidationError` in addition to `ValueError` — non-UUID strings (e.g. empty string or garbage) would previously blow up with an unhandled 500. In-scope sub-issue, fixed in this task.
- Added `backend/apps/accounts/tests/test_invite_accept.py` — 12 integration tests covering GET (valid, unknown, invalid-string, missing, cancelled, already-used) and POST (happy path ×2, already-accepted, unknown token, missing password, weak password). All 12 pass.
- Updated `content/emails/invite_tenant.md` with `invite_url_pattern` note documenting the new `/invite/{token}` URL scheme.
