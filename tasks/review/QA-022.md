## Goal
Fix the pytest-django fixture lifecycle issue where the test database is destroyed after the first TestCase class tearDown, causing all tests in the second class to error with `database "test_klikk_db" does not exist`.

2026-04-24 — Promoted from discovery `2026-04-24-pytest-django-class-teardown.md`. First class (14 tests) passes; second class (6 tests) errors on DB setup. Both inherit `TremlyAPITestCase`.

2026-04-25 — Two root causes found and fixed:

**Bug 1 — Stale DB teardown race**: When `test_klikk_db` was left over from a crashed prior run, pytest-django's `setup_databases()` (session-scoped) destroyed and recreated the DB.  Django's `TestCase.tearDownClass()` closes all connections after the first class, so the second class's `setUpClass()` tried to enter an atomic block on a connection pointing at the DB mid-recreation, resulting in `FATAL: database "test_klikk_db" does not exist`.  Fix: added `--reuse-db` to `pytest.ini` `addopts` so the DB is never destroyed between runs; use `--create-db` explicitly after migration changes.

**Bug 2 — DRF browsable API HTML instead of JSON**: `APITestCase.client` sends `Accept: */*` by default, causing DRF to return the browsable HTML renderer.  All endpoint tests that called `resp.json()` failed with `ValueError: Content-Type header is "text/html"`.  Fix: added `_JsonAPIClient` subclass to `apps/test_hub/base/test_case.py` that overrides `request()` to inject `Accept: application/json` as a default.  Set as `TremlyAPITestCase.client_class` so all subclasses inherit it without needing `super().setUp()` calls.

Verified: 20/20 pass on fresh DB (no pre-existing `test_klikk_db`) and on second run (reuse). 780 existing `apps/test_hub` tests pass without regression. `backend/TESTING.md` created to document both patterns.
