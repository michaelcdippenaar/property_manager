## Goal

Fix the validation guardrail override flow so that confirming an override (with a reason) reliably unlocks the Send for Signing button and allows the e-signing flow to proceed.

## Acceptance criteria

- [x] Reproduce locally and capture the override-confirm event flow with Vue devtools.
- [x] Identify whether the override is failing to set, failing to propagate, or failing to invalidate the Send-button visibility computed property.
- [x] Fix: override + reason + OK reliably unlocks the Send for Signing button.
- [x] Audit log: every override persists a record (actor, lease ID, reason, timestamp); confirm it exists and add it if missing.
- [x] Add a Vitest browser test simulating the full override flow.
- [ ] Verify end-to-end: open lease with failing validation → override → enter reason → OK → button appears → click → submission proceeds.

## Files likely touched

- `admin/src/views/leases/ESigningPanel.vue` (owner of Send button + override modal trigger)
- `admin/src/components/leases/ValidationOverrideModal.vue` (or wherever "override" / "guardrail" modal lives — search `override` + `guardrail`)
- `admin/src/stores/leases.ts` (if override flag is stored in Pinia)
- `backend/apps/leases/views.py` (audit log endpoint if missing)

## Root cause hypotheses

1. Override state stored in modal component but not propagated to the parent that owns the Send button visibility.
2. Modal closes via `cancel()` path instead of `confirm()` path, so the override flag never sets.
3. Override flag is set on the wrong store / scope (local `ref` vs Pinia store).
4. Send button visibility is computed before the override flag is set; computed property is not reactive to the override flag.

## Test plan

**Manual:**
1. Open any lease that has unmet validation requirements.
2. Attempt Send for Signing — verify guardrail dialog appears.
3. Click "Override for testing" (or equivalent), enter a reason, click OK.
4. Confirm Send for Signing button is now enabled.
5. Click the button and confirm the e-signing flow proceeds normally.

**Automated:**
- `cd admin && npx vitest run` — new browser test covering override flow in `ESigningPanel`.

2026-04-26 — rentals-implementer: Root cause confirmed as hypothesis 4 variant — the "empty state / Send for Signing" block in ESigningPanel.vue used `v-else-if` chained directly to the `v-if="rhaOverride"` override-banner block. When `rhaOverride` became truthy the banner rendered but the `v-else-if` branch was unconditionally skipped by Vue's directive chain, leaving the Send button hidden even after a successful override. No separate state propagation failure.

Fix: changed `v-else-if="leaseData?.status !== 'active'"` to a standalone `v-if="!latestSub && leaseData?.status !== 'active'"` with an explanatory comment. The Send button's own `:disabled` guard (`rhaBlockingFlags.length > 0 && !rhaOverride`) was already correct — it only needed to be visible.

Audit log: `Lease.rha_override` JSON field in `backend/apps/leases/models.py` stores `user_id`, `user_email`, `reason`, `overridden_at`, and `flags_at_override`. All required audit fields are present; no new model/migration needed.

New test: `admin/src/views/leases/__tests__/esigning-override.browser.test.ts` — 5 test cases covering: button visible with no flags, button hidden with blocking flags + no override, button visible + enabled after override set (core regression), full submitOverride API call flow, and audit object shape.
