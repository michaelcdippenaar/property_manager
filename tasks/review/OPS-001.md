## Goal
Stand up a CI/CD pipeline so every PR is linted/tested automatically and every main-branch merge deploys to staging. Without this, regressions land silently and deploys are manual + error-prone.

## Acceptance criteria
- [x] `.github/workflows/ci.yml`: runs on PR — black/ruff, eslint, `pytest backend`, `npm test` for admin + web_app, `flutter analyze`
- [x] `.github/workflows/staging-deploy.yml`: on merge to main → build Docker images, push to registry, SSH + `docker compose pull && up -d` on staging host
- [ ] Branch protection: PR requires green CI to merge
- [x] Status badge in repo README
- [x] Secrets managed as GitHub Actions secrets (SSH key, registry creds, env)
- [x] Rollback: previous image tag kept, documented one-command rollback

## Files likely touched
- `.github/workflows/ci.yml` (new)
- `.github/workflows/staging-deploy.yml` (new)
- `backend/Dockerfile`, `admin/Dockerfile`, `web_app/Dockerfile`
- `docs/ops/ci-cd.md` (new)

- Trigger rollback → previous SHA live

### 2026-04-22 — reviewer: Review requested changes

Two issues must be fixed before this can proceed to testing. One is a functional blocker that will cause every CI pytest run to fail; the other is a security concern.

**1. CI pytest will fail — Postgres DB name mismatch (blocker)**

File: `.github/workflows/ci.yml`, `backend` job, `services.postgres` block and `Run pytest` step.

`backend/config/settings/base.py` resolves database config via `decouple.config("DB_NAME", default="klikk_db")` — it does NOT read `DATABASE_URL`. The CI service container is created with `POSTGRES_DB: klikk_test`. The pytest step injects `DATABASE_URL: postgres://klikk_user:klikk_pass@localhost:5432/klikk_test` which settings never consumes.

Result: Django connects to `klikk_db` (the default), which does not exist in the container, and pytest fails with a database-does-not-exist error before running a single test.

Fix: In the `Run pytest` step env block, replace (or supplement) `DATABASE_URL` with the individual decouple vars that `base.py` actually reads:

```yaml
env:
  DJANGO_SETTINGS_MODULE: config.settings.local
  DB_NAME: klikk_test
  DB_USER: klikk_user
  DB_PASSWORD: klikk_pass
  DB_HOST: localhost
  DB_PORT: "5432"
  SECRET_KEY: ci-test-secret-key-not-used-in-production
  DEBUG: "True"
  ALLOWED_HOSTS: "localhost,127.0.0.1"
```

Also remove the `DATABASE_URL` line — it is unused and misleading.

**2. GHCR_PAT interpolated directly into SSH heredoc (security)**

File: `.github/workflows/staging-deploy.yml`, `Deploy via SSH` step, line ~106.

```yaml
echo "${{ secrets.GHCR_PAT }}" | docker login ghcr.io -u "${GHCR_OWNER}" --password-stdin
```

`${{ secrets.GHCR_PAT }}` is expanded by the Actions runner before the heredoc text is sent over SSH. This embeds the PAT value as a literal string in the SSH session payload. GitHub's log masking covers runner stdout but does not protect the secret once it is embedded in a heredoc transmitted to a remote host. This also makes the workflow vulnerable to heredoc injection if any other variable in the same block were ever sourced from untrusted input.

Fix: Pass the PAT as an SSH environment variable instead of interpolating it inline. Use `ssh -o SendEnv=GHCR_PAT` (requires `AcceptEnv GHCR_PAT` in sshd_config on the server), or use the `appleboy/ssh-action` action with an `envs:` field — it passes secrets as env vars to the remote session rather than embedding them in the command string. Example using appleboy:

```yaml
- name: Deploy via SSH
  uses: appleboy/ssh-action@v1
  with:
    host: ${{ secrets.STAGING_HOST }}
    username: ${{ secrets.STAGING_SSH_USER }}
    key: ${{ secrets.STAGING_SSH_KEY }}
    envs: IMAGE_TAG,GHCR_OWNER,GHCR_PAT
    script: |
      set -e
      ...
      echo "$GHCR_PAT" | docker login ghcr.io -u "$GHCR_OWNER" --password-stdin
      ...
```

If the inline SSH approach is kept, the PAT must be passed via `-o SendEnv` and never embedded in the heredoc via `${{ }}` substitution.

`website_web` in `deploy/docker-compose.staging.yml` has no `image:` override and is not built or pushed by the CI/CD workflow.
