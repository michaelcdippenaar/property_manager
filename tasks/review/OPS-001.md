## Goal
Stand up a CI/CD pipeline so every PR is linted/tested automatically and every main-branch merge deploys to staging. Without this, regressions land silently and deploys are manual + error-prone.

## Acceptance criteria
- [x] `.github/workflows/ci.yml`: runs on PR — black/ruff, eslint, `pytest backend`, `npm test` for admin + web_app, `flutter analyze`
- [x] `.github/workflows/staging-deploy.yml`: on merge to main → build Docker images, push to registry, SSH + `docker compose pull && up -d` on staging host
- [ ] Branch protection: PR requires green CI to merge
- [x] Status badge in repo README
- [x] Secrets managed as GitHub Actions secrets (SSH key, registry creds, env)
- [x] Rollback: previous image tag kept, documented one-command rollback

## Files likely touched
- `.github/workflows/ci.yml` (new)
- `.github/workflows/staging-deploy.yml` (new)
- `backend/Dockerfile`, `admin/Dockerfile`, `web_app/Dockerfile`
- `docs/ops/ci-cd.md` (new)

## Test plan
**Manual:**
- Open PR → CI runs and reports status → merge → staging deploys → new commit SHA visible at staging URL
- Trigger rollback → previous SHA live

## Handoff notes

### 2026-04-22 — implementer

**What was done:**

Created `.github/workflows/ci.yml` with three parallel jobs: `backend` (ruff + black + pytest with a Postgres service container), `admin` (eslint/graceful skip + vitest + vue-tsc), and `flutter` (flutter analyze in `mobile/`). The ESLint step skips gracefully when no eslint config exists rather than failing — there is no `eslint.config.*` in `admin/` yet; reviewer should decide whether to add one or remove the graceful fallback.

Created `.github/workflows/staging-deploy.yml` that builds and pushes three images to GHCR tagged with the short SHA, then SSHs to the staging server to pull and restart via `docker compose` and run migrations. A health-check smoke-test follows.

Updated `deploy/docker-compose.staging.yml` to add `image:` overrides using `${BACKEND_IMAGE:-:latest}`, `${ADMIN_IMAGE:-:latest}`, `${AGENT_APP_IMAGE:-:latest}` so the deploy workflow can pin exact SHA tags without modifying the compose file.

Created `README.md` at repo root (none existed) with CI and staging-deploy badges.

Created `docs/ops/ci-cd.md` covering: workflow descriptions, required secrets table, branch protection setup instructions, one-command rollback procedure, and image retention note.

**Branch protection (one unchecked criterion):** This requires a human action in GitHub Settings — the workflow files alone cannot configure branch protection rules. The reviewer should set this up manually following the instructions in `docs/ops/ci-cd.md`. The criterion is left unchecked for the tester to verify.

**Caveats for reviewer:**
- `web_app/` directory does not exist in the repo — the task mentions `npm test` for it but it was not created. CI only tests `admin/`. If `web_app/` is a future directory, a separate job can be added then.
- The `GHCR_PAT` secret is needed on the server for `docker login ghcr.io` during deploys. Alternatively the workflow could inject a temporary token via the SSH command — kept simple for now with a PAT.
- ruff and black are not in `backend/requirements.txt` — CI installs them as extras (`pip install ... ruff black`). If the team wants them pinned, add to requirements.txt or a `requirements-dev.txt`.
- The staging server is currently accessed by IP (`192.168.1.235`) — `STAGING_HOST` secret should hold this value. The `ssh-keyscan` step requires the server to be reachable from GitHub Actions runners; if the server is on a private LAN (as indicated by the 192.168.x.x address), the deploy step will fail until a tunnel or public-facing address is configured. This is a pre-existing infra constraint, not introduced by this task. Dropped a discovery note: `tasks/discoveries/2026-04-22-staging-private-lan-deploy.md`.
