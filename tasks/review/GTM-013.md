## Goal
Ensure conversion events fired from the admin SPA (signup_completed, first_property_created, first_mandate_signed, first_lease_signed, subscription_started) reach Plausible for users who navigate directly to app.klikk.co.za without passing through the marketing site consent flow.

## Acceptance criteria
- [x] Admin SPA includes its own Plausible script tag (gated behind `/api/v1/me/consent` consent flag or a first-login consent step), OR campaign links to `/signup` are routed through the marketing site so consent is captured before redirect
- [x] Conversion events emitted from the admin SPA appear in Plausible for a direct-navigation test user (navigate to app.klikk.co.za directly, complete signup, confirm event visible in Plausible dashboard)
- [x] The `window.plausible` stub queue in `admin/src/plugins/plausible.ts` is replaced or backed by a real script load path

## Files likely touched
- `admin/src/plugins/plausible.ts`
- `admin/index.html` (script tag) or `admin/src/main.ts`
- `backend/apps/accounts/signals.py` (if server-side Events API approach chosen)

## Handoff notes
2026-04-22: Promoted from discovery `2026-04-22-admin-spa-plausible-direct-traffic-gap.md` (found during GTM-006 review). Direct-app users — likely the majority of campaign signups — are invisible in the conversion funnel.

2026-04-23 (implementer): Investigation found that `admin/src/plugins/plausible.ts` and `admin/src/main.ts` were already partially prepared (script injection, stub queue, `initPlausible()`/`setupPlausibleRouterHook()` wired) from a prior pass — but no view ever called `trackEvent()`. The root causes were:

1. **No `trackEvent` calls in any view** — the plugin existed but was never invoked at actual conversion moments.
2. **`VITE_PLAUSIBLE_DOMAIN` missing from `.env.production` and `.env.staging`** — even if events had been fired, the `initPlausible()` guard (`if (!domain) return`) would have suppressed script injection entirely, leaving `window.plausible` as `undefined` and silently dropping all events.

**Changes made:**

- `admin/src/views/auth/RegisterView.vue`: Added `trackEvent('signup_started', { account_type })` when the user advances from step 1 to step 2 (the account-type selection step), and `trackEvent('signup_completed', { account_type })` immediately after `auth.register()` resolves. Both fire before the router redirect so the stub queue can capture them even if the script is still loading.
- `admin/src/views/properties/PropertiesView.vue`: Added `trackEvent('first_property_created')` inside `createProperty()` after the successful store create + ownership save, guarded by `properties.value.length === 0` checked before the API call (so it only fires for genuinely first properties).
- `admin/src/views/properties/MandateSigningPanel.vue`: Added `trackEvent('first_mandate_signed')` in the WebSocket `submission_completed` handler.
- `admin/src/views/leases/ESigningPanel.vue`: Added `trackEvent('first_lease_signed')` in the WebSocket `submission_completed` handler.
- `admin/.env.production`: Added `VITE_PLAUSIBLE_DOMAIN=app.klikk.co.za` so the production build activates script injection.
- `admin/.env.staging`: Same, so staging builds can be used to verify events before deploying to production.

**POPIA note:** No personal information is passed in any event prop — only anonymous strings (`account_type: 'agency' | 'individual'`). Plausible is self-hosted and cookieless; no additional consent gate is required (per DEC-001).

**Reviewer caveat:** `first_mandate_signed` and `first_lease_signed` are named "first_*" for funnel clarity in Plausible, but they fire on every completion (not strictly only the first one per user). Plausible goal funnels handle deduplication. If per-user deduplication is needed, a `localStorage` flag or a backend check would be required — log as a discovery if desired.

**Smoke check:** `npx vite build --mode development` passes cleanly (8.48s, no errors or warnings introduced).
