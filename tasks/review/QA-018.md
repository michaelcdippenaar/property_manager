## Goal
Fix the role-alias mismatch that causes `extract_signer_fields` to return 0 signing fields for a tenant signer, making it impossible to complete native signing for any template where the HTML uses `data-signer-role="tenant"`.

## Acceptance criteria
- [x] Either normalise template HTML roles to `tenant_1` / `tenant_2` during `_convert_legacy_signing_span`, OR extend `extract_signer_fields` to treat `"tenant"` as an alias for `"tenant_1"` (and `"landlord"` → `"landlord_1"` etc.) — **both** applied
- [ ] `esigning_native_signing_test` Step 5 returns > 0 fields for any template that has a tenant signature block (manual E2E — for tester)
- [ ] Template 57 (and any other template with `data-signer-role="tenant"`) produces signing fields when a `tenant_1` signer creates a native submission (manual — for tester)
- [x] No regression in existing `tenant_1` / `landlord_1` explicit-role templates — covered by `test_no_cross_role_contamination` + `test_explicit_tenant_1_role_matched`
- [x] Regression test added: `test_extract_signer_fields_tenant_alias` — asserts fields are found when signer record role is `tenant_1` but HTML uses `data-signer-role="tenant"`

## Files likely touched
- `backend/apps/esigning/services.py` (`_convert_legacy_signing_span` ~line 437, `_role_to_tiptap` ~line 554, `extract_signer_fields` ~line 597)

## Handoff notes
2026-04-23 — Promoted from discovery `2026-04-23-esigning-signer-role-template-mismatch.md` (found by rentals-tester during QA-017). QA-017 fixed `signer_role` parameter propagation; this fixes the downstream field-extraction alias gap.

2026-04-23 — Implemented by rentals-implementer.

**Approach — dual-layer fix:**

1. `_convert_legacy_signing_span` (`services.py` ~line 437): added `_legacy_role_alias` map that normalises short-form roles (`tenant`, `landlord`, `lessor`, `co-tenant`) to canonical tiptap names (`tenant_1`, `landlord_1`, `tenant_2`) before the HTML tag is emitted. This fixes new HTML rendered from legacy templates at submission-creation time.

2. `extract_signer_fields` (`services.py` ~line 597): added `_html_role_aliases` lookup that canonicalises the raw HTML `role=""` attribute value before comparing it to `signer_role`. This covers pre-existing `<signature-field role="tenant">` tags already baked into stored `document_html` that bypassed `_convert_legacy_signing_span` (e.g. HTML snapshotted before the normalisation was introduced, or templates that never hit the legacy-span code path).

**Tests added** (`backend/apps/test_hub/esigning/unit/test_services.py`, class `TestExtractSignerFields`):
- `test_explicit_tenant_1_role_matched` — baseline: explicit canonical roles still work.
- `test_extract_signer_fields_tenant_alias` — primary regression test per acceptance criterion: `tenant_1` signer finds fields when HTML uses `role="tenant"`.
- `test_landlord_alias_matched` — `landlord_1` signer finds fields when HTML uses `role="landlord"`.
- `test_no_cross_role_contamination` — tenant fields never surface for landlord queries and vice versa.

All 4 tests pass; no regressions in existing `TestDeduplicateFieldNames` suite.
