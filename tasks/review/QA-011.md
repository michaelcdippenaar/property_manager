## Goal
Trap keyboard focus inside modals and drawers, and fix the invalid `<label for>` association in AcceptInviteView, so keyboard-only and screen-reader users are not stranded.

## Acceptance criteria
- [x] On modal/drawer `open`, focus moves to the first focusable element inside the dialog
- [x] Tab cycles within the dialog only; Shift+Tab wraps to the last element
- [x] Escape closes the dialog and returns focus to the element that triggered it
- [x] Implemented via `focus-trap` npm composable or native `<dialog>` element in `BaseModal.vue` and `BaseDrawer.vue`
- [ ] Tester validates with keyboard-only navigation and VoiceOver
- [x] In `AcceptInviteView.vue` lines 48–49: replace `<p id="invite-email">` with `<input type="email" id="invite-email" :value="invite.email" readonly tabindex="-1">` so the `<label for>` association is valid per spec
- [ ] All existing a11y tests still pass; no new console a11y warnings

### 2026-04-22 — implementer

**Focus trap (BaseModal + BaseDrawer):**
Created `admin/src/composables/useFocusTrap.ts` — a zero-dependency composable (no new npm package required). It:
- Captures the previously-focused element on `activate()` and restores it on `deactivate()`.
- Moves focus to the first focusable child immediately on activation.
- Listens for `keydown` on `document`: Escape emits `close`; Tab/Shift+Tab wrap within the dialog's focusable elements (computed via a standard CSS selector list excluding `tabindex="-1"` and `inert` subtrees).

Both `BaseModal.vue` and `BaseDrawer.vue` wire `activate` to `<Transition @after-enter>` and `deactivate` to `<Transition @before-leave>`, so focus is trapped only while the panel is fully rendered. The `trapRef` is bound to the dialog panel `<div>`.

**AcceptInviteView.vue:**
Replaced `<p id="invite-email">` with `<input type="email" id="invite-email" :value="invite.email" readonly tabindex="-1" aria-readonly="true">`. This creates a valid labelable element per HTML spec while remaining non-interactive for the user (readonly + tabindex="-1" prevents it from entering the normal Tab order and accepting edits).

**Type-check:** `npx vue-tsc --noEmit` shows only pre-existing errors across unrelated files (deleted components, ImportMeta.env issues). Zero new errors from these three files.
